---
title: Socket API
description: Control a running Gmux server from scripts and tools.
---

Gmux exposes a local socket API for scripts and tools that need to inspect or control a running session.

Most automation should start with the CLI wrappers. Use the raw socket API only when you need direct request/response control or long-lived event subscriptions.

## Choose a control layer

| Layer | Use it for |
| --- | --- |
| CLI wrappers | Shell scripts, simple orchestration, and human debugging. |
| Raw socket API | Custom tools, protocol clients, and event subscribers. |

The layers share the same control surface.

## What you can control

The socket API can:

- create, list, focus, rename, and close tabs
- list, inspect, focus, resize, split, rename, read, close, and send input to panes
- subscribe to events and wait for output
- stop the server and reload config

## CLI examples

Create a tab:

```bash
gmux new-tab -n logs
```

Split a pane and run a command:

```bash
gmux split-pane -h "npm test"
```

Send input:

```bash
gmux send-keys -t 1-2 C-c
```

Read pane output:

```bash
gmux pane read 1-2 --source recent --lines 50
```

## Raw methods

Raw socket method names use dot notation:

| Area | Methods |
| --- | --- |
| Server | `ping`, `server.stop`, `server.reload_config` |
| Tab | `tab.create`, `tab.list`, `tab.get`, `tab.focus`, `tab.rename`, `tab.close` |
| Pane | `pane.split`, `pane.list`, `pane.get`, `pane.focus`, `pane.resize`, `pane.rename`, `pane.send_text`, `pane.send_keys`, `pane.send_input`, `pane.read`, `pane.close`, `pane.wait_for_output` |
| Events | `events.subscribe`, `events.wait` |

## Socket transport

Gmux uses newline-delimited JSON over a Unix domain socket.

Send one request per line:

```json
{"id":"req_1","method":"ping","params":{}}
```

A successful response includes the same `id`:

```json
{"id":"req_1","result":{"type":"pong"}}
```

Event subscriptions keep the connection open after the initial response.

## Socket paths

The default socket lives under your Gmux config directory.

Named sessions have separate sockets:

```text
~/.config/gmux/gmux.sock
~/.config/gmux/sessions/<name>/gmux.sock
```

Resolution order:

1. explicit CLI `--session <name>`
2. `GMUX_SOCKET_PATH`
3. `GMUX_SESSION=<name>`
4. default session socket

Use `GMUX_SOCKET_PATH` only for low-level overrides.

## Event subscriptions

Subscribe to events when you need a long-lived stream:

The first response acknowledges the subscription. Later lines are pushed events.

Use `events.wait` when you want one matching event and then a response.

## Reading panes

Use `pane.read` through the CLI unless you are writing a protocol client.

```bash
gmux pane read p_1 --source visible --lines 80
gmux pane read p_1 --source recent --lines 120
gmux pane read p_1 --source recent-unwrapped --lines 120
```

`recent-unwrapped` is useful for logs because it ignores soft wrapping.

## Waiting

Use waits to coordinate scripts with pane output.

```bash
gmux wait output p_1 --match "server ready" --timeout 10000
```

## Response shapes

Successful responses look like this:

```json
{
  "id": "req_1",
  "result": {
    "type": "pane_info",
    "pane": {
      "pane_id": "p_1",
      "terminal_id": "term_abc123",
      "tab_id": "t_1",
      "focused": true,
      "revision": 42
    }
  }
}
```

Errors look like this:

```json
{
  "id": "req_1",
  "error": {
    "code": "not_found",
    "message": "pane not found"
  }
}
```

## Protocol stability

Gmux has a protocol version for client/server compatibility. Protocol changes are reviewed with release compatibility in mind.

Check the server protocol with `ping` or `gmux status` before depending on new behavior. Handle unknown fields gracefully.
