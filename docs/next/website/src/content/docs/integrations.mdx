Gmux detects supported agents automatically. Official integrations can add semantic state reports for agent-oriented workflows while the public Gmux UI continues to focus on panes and sessions.

Use integrations when you need direct state reports from Pi/OMP/Copilot/Kimi/Hermes/Qoder-style hooks or plugins.

## Install integrations

Open settings inside Gmux and use the integrations tab to install recommended integrations for agents found on your `PATH`, or run commands manually:

```bash
gmux integration install pi
gmux integration install omp
gmux integration install claude
gmux integration install codex
gmux integration install copilot
gmux integration install droid
gmux integration install kimi
gmux integration install opencode
gmux integration install hermes
gmux integration install qodercli
```

## Uninstall integrations

```bash
gmux integration uninstall pi
gmux integration uninstall omp
gmux integration uninstall claude
gmux integration uninstall codex
gmux integration uninstall copilot
gmux integration uninstall droid
gmux integration uninstall kimi
gmux integration uninstall opencode
gmux integration uninstall hermes
gmux integration uninstall qodercli
```

## How Gmux uses integrations

Gmux combines three legacy agent signals:

| Signal | Purpose |
| --- | --- |
| Process detection | Identifies which pane owns which running process. |
| Screen heuristics | Infers visible native agent state such as `working` or `blocked`. |
| Integration events | Reports semantic state for integrations that still provide it. |

Claude Code, Codex, Droid, and OpenCode state detection is screen-read first. Their Gmux-owned integrations do not author `idle`, `working`, or `blocked` state. Pi, GitHub Copilot CLI, OMP, Kimi Code CLI, Hermes Agent, and Qoder CLI can report semantic state. Custom socket integrations can report state when they define state that is not visible in the native terminal UI.

## Pi

Install the Pi integration:

```bash
gmux integration install pi
```

Gmux writes the bundled extension to:

```text
~/.pi/agent/extensions/gmux-agent-state.ts
```

If `PI_CODING_AGENT_DIR` is set, Gmux writes to `$PI_CODING_AGENT_DIR/extensions/gmux-agent-state.ts` instead. The extensions directory must already exist. Uninstall removes only that extension file.

## OMP

Install the OMP integration:

```bash
gmux integration install omp
```

Gmux writes the bundled extension to:

```text
~/.omp/agent/extensions/gmux-omp-agent-state.ts
```

If `PI_CODING_AGENT_DIR` is set, Gmux writes to `$PI_CODING_AGENT_DIR/extensions/gmux-omp-agent-state.ts` instead. The extensions directory must already exist. Uninstall removes only that extension file.

The OMP integration reports `omp` as the agent label through Gmux's socket API. It does not require native process detection for the `omp` executable.

## Claude Code

Install the Claude Code hook:

```bash
gmux integration install claude
```

The hook reports Claude Code session identity to the local Gmux socket on session start. Claude Code state comes from Gmux's screen detection.

Gmux uses `~/.claude` by default, or `CLAUDE_CONFIG_DIR` when set. The Claude config directory must already exist. Install writes `hooks/gmux-agent-state.sh` and updates `settings.json` with Gmux hook entries. Uninstall removes the matching hook entries and deletes the hook script.

## Codex

Install the Codex hook:

```bash
gmux integration install codex
```

The Codex hook reports session identity through the same local socket API used by other integrations. Codex state comes from Gmux's screen detection.

Gmux uses `~/.codex` by default, or `CODEX_HOME` when set. The Codex config directory must already exist. Install writes `gmux-agent-state.sh`, updates `hooks.json`, and ensures `[features] hooks = true` in `config.toml`. It also removes the deprecated top-level `codex_hooks` flag when present. Uninstall removes Gmux entries from `hooks.json` and deletes the hook script, but leaves `config.toml` unchanged.

## GitHub Copilot CLI

Install the GitHub Copilot CLI hook:

```bash
gmux integration install copilot
```

Copilot state is reported through the same local socket API used by other integrations.

Gmux uses `~/.copilot` by default, or `COPILOT_HOME` when set. The Copilot config directory must already exist. Install writes `hooks/gmux-agent-state.sh` and updates `settings.json` with Gmux hook entries. The hook maps prompt submission, tool use, post-tool progress after approvals, `ask_user` prompts, `exit_plan_mode` plan review prompts, permission notifications, agent idle notifications, turn stops, and session exits into Gmux state. Uninstall removes Gmux entries from `settings.json` and deletes the hook script.

After Copilot emits a session-bearing event, Gmux can use the reported session id to resume the pane with `copilot --resume=<id>`.

## Kimi Code CLI

Install the Kimi Code CLI hook:

```bash
gmux integration install kimi
```

The hook reports semantic state to Gmux when Kimi Code emits lifecycle, tool, approval, and session events. It requires Kimi Code CLI `0.8.0` or newer.

Gmux uses `~/.kimi-code` by default, or `KIMI_CODE_HOME` when set. The Kimi Code config directory must already exist. Install writes `hooks/gmux-agent-state.sh` and appends Gmux-managed `[[hooks]]` entries to `config.toml`. Uninstall removes the Gmux-managed config block and deletes the hook script.

Native screen heuristics remain available when the hook is not installed.

## Droid

Install the Droid hook:

```bash
gmux integration install droid
```

The Droid hook reports session identity through the same local socket API used by other integrations. Droid state comes from Gmux's screen detection.

Gmux uses `~/.factory` for Droid hooks. The Factory config directory must already exist. Install writes `hooks/gmux-agent-state.sh`, updates `settings.json` with a Gmux `SessionStart` hook entry, and removes older Gmux Droid hook entries from `hooks.json` if present. Uninstall removes Gmux entries from both config files and deletes the hook script.

After Droid emits a session start event, Gmux can use the reported session id to resume the pane with `droid --resume <id>`.

## OpenCode

Install the OpenCode plugin:

```bash
gmux integration install opencode
```

Gmux writes the plugin to `~/.config/opencode/plugins/gmux-agent-state.js`. The OpenCode config directory must already exist. Uninstall removes only that plugin file.

The plugin reports session identity while OpenCode runs inside a Gmux pane. After OpenCode emits a session-bearing event, Gmux can use the reported session id to resume the pane with `opencode --session <id>`. OpenCode state comes from Gmux's screen detection.

## Hermes Agent

Install the Hermes Agent plugin:

```bash
gmux integration install hermes
```

Gmux writes `~/.hermes/plugins/gmux-agent-state/` and enables `gmux-agent-state` in `~/.hermes/config.yaml`. The Hermes config directory must already exist. Restart Hermes after installing so the plugin loads. Uninstall removes the plugin directory and removes `gmux-agent-state` from `plugins.enabled`.

The plugin reports lifecycle, tool, approval state, and session id while Hermes runs inside a Gmux pane. Gmux can use the reported session id to resume the pane with `hermes --resume <id>`. Native screen heuristics remain available when the plugin is not installed.

## Qoder CLI

Install the Qoder CLI hook:

```bash
gmux integration install qodercli
```

The hook reports semantic state to Gmux when Qoder CLI emits lifecycle events.

Gmux uses `~/.qoder` by default, or `QODER_CONFIG_DIR` when set. The Qoder config directory must already exist. Install writes `hooks/gmux-agent-state.sh` and updates `settings.json` with Gmux hook entries. Uninstall removes the matching hook entries and deletes the hook script.

Native screen heuristics remain available when the hook is not installed.

## Custom status labels

Integrations can report a short visual label without changing the semantic state.

For example, an agent can remain semantically `working` while showing `indexing` in the UI.

```bash
gmux pane report-agent 1-1 \
  --source custom:docs \
  --agent docs-bot \
  --state working \
  --custom-status indexing
```

User hooks that run next to a Gmux-managed integration should use metadata instead of `report-agent`. Metadata changes presentation without taking over the integration's `idle`, `working`, `blocked`, or session restore authority. `--agent` guards the report so it only applies while that authoritative agent is active. `--applies-to-source` guards the report so it only applies while that lifecycle authority source is active. `--display-agent` changes the visible name.

```bash
gmux pane report-metadata "$GMUX_PANE_ID" \
  --source user:claude-title \
  --agent claude \
  --title "Refactor auth middleware" \
  --display-agent "Claude: auth" \
  --custom-status "refactor auth" \
  --state-label working="refactoring auth" \
  --ttl-ms 3600000
```

Custom status and state labels are visual-only. Waits, notifications, and workspace rollups still use the semantic state.

## Debug integration state

List known agents:

```bash
gmux agent list
```

Read a pane when you need to verify what Gmux can see:

```bash
gmux pane read 1-1 --source recent --lines 50
```

If integration state looks wrong, first confirm the agent is running inside Gmux and that the relevant hook or plugin was installed for the same user account.
